# Three Editor by AI 宪法原则

**版本**: 1.0.0  
**生效日期**: 2026-04-07  
**最后修订**: 2026-04-07

---

## 前言

本宪法定义了 Three Editor by AI 项目的核心原则、设计哲学和开发规范。所有代码、文档和决策必须符合本宪法规定的原则。

---

## 核心原则

### I. 用户至上 (User First)

**所有功能设计必须以用户体验为核心**

- 任何功能的添加都必须有明确的用户价值
- 交互设计必须直观、一致、可预测
- 性能优化优先保障用户感知性能（首屏加载、交互响应）
- 错误提示必须友好、清晰、可操作

**违反示例**:
- 添加用户不需要的复杂功能
- 交互模式不一致（相同操作不同行为）
- 无反馈的长时间操作

---

### II. 模块化与职责分离 (Modularity & Separation of Concerns)

**代码必须按职责清晰分层，模块间保持松耦合**

- UI 层：仅负责界面展示和用户交互
- 逻辑层（Composables）：封装业务逻辑和状态管理
- 引擎层（Core）：封装 Three.js 核心逻辑
- 服务层：封装外部 API 和数据访问
- 工具层：提供通用工具函数

**模块通信规则**:
- 上层可调用下层，下层不可依赖上层
- 跨模块通信使用事件总线（mitt）
- 禁止循环依赖

**违反示例**:
- 在 Vue 组件中直接操作 Three.js 对象
- Core 层依赖 UI 组件
- 模块间直接访问内部状态而非通过事件

---

### III. Composition API 优先 (Composition API First)

**所有新代码必须使用 Vue 3 Composition API**

- 使用 `<script setup>` 语法
- 逻辑复用通过 composables 实现
- 避免使用 Options API（除非维护旧代码）
- 响应式状态使用 `ref`、`reactive`、`computed`

**命名规范**:
- Composable 函数：`useXxx` 格式（如 `useThreeViewer`）
- 响应式变量：清晰描述性名称
- 计算属性：使用 `computed` 明确标识

**违反示例**:
- 在新组件中使用 `data()`、`methods`、`mounted`
- 逻辑复用使用 mixins 而非 composables

---

### IV. 事件驱动解耦 (Event-Driven Decoupling)

**核心管理器必须使用事件总线实现模块解耦**

- ThreeViewer、ObjectManager、InputManager、AssetLoader 必须集成 mitt
- 状态变更必须触发相应事件
- 监听器必须及时清理，避免内存泄漏

**事件命名规范**:
- 格式：`[entity-]action[-status]`
- 示例：`object-added`、`scene-loaded`、`load-error`

**违反示例**:
- 直接调用其他模块的内部方法
- 状态变更不触发事件
- 未清理的监听器

---

### V. 序列化完整性 (Serialization Integrity)

**场景序列化必须完整、一致、可恢复**

- 用户数据（userData）必须完整序列化
- 运行时对象（如 _mixer）不得序列化
- 反序列化后场景状态必须与原始状态一致
- 动画状态必须可恢复

**序列化边界**:
- ✅ 序列化：几何体、材质、位置、旋转、缩放、userData、动画索引
- ❌ 不序列化：运行时临时对象、缓存、事件监听器

**违反示例**:
- userData 中包含运行时对象引用
- 反序列化后动画无法播放
- 序列化数据包含不可 JSON 序列化的对象

---

### VI. 性能意识 (Performance Awareness)

**所有代码必须考虑性能影响**

- 避免不必要的重复计算（使用缓存、记忆化）
- 避免阻塞主线程的长时间操作
- 及时清理不需要的对象和监听器
- 资源加载必须实现缓存机制

**性能红线**:
- 帧率不得低于 30 FPS（复杂场景）
- 交互响应延迟不得超过 100ms
- 首屏加载时间不得超过 3 秒

**违反示例**:
- 在渲染循环中执行昂贵操作
- 重复加载已缓存的资源
- 未清理的事件监听器导致内存泄漏

---

### VII. 渐进增强 (Progressive Enhancement)

**功能实现采用渐进式策略**

- 核心功能优先（场景渲染、对象操作）
- 增强功能逐步添加（后处理、动画）
- 可选功能支持优雅降级
- 新功能必须有明确的 MVP 范围

**优先级**:
1. P0: 场景渲染、对象创建/变换、基础材质
2. P1: 资源管理、场景序列化、相机控制
3. P2: 动画系统、后处理、高级材质
4. P3: 辅助功能、美化、优化

**违反示例**:
- 在核心功能未完成时添加边缘功能
- 新功能影响现有功能稳定性

---

### VIII. 文档驱动 (Documentation Driven)

**代码变更必须伴随文档更新**

- 新增功能必须更新 README.md
- API 变更必须更新 API.md
- 架构调整必须更新 ARCHITECTURE.md
- 复杂逻辑必须添加代码注释

**注释规范**:
- JSDoc 格式注释公共 API
- 复杂算法必须添加实现思路注释
- 魔法数字必须添加注释说明

**违反示例**:
- 新增 API 不更新文档
- 复杂逻辑无任何注释
- 文档与实际代码不一致

---

### IX. 代码质量 (Code Quality)

**代码必须清晰、简洁、可维护**

- 函数长度不超过 50 行
- 文件长度不超过 500 行（工具函数除外）
- 变量和函数名称必须自解释
- 避免嵌套过深（不超过 3 层）

**代码审查要点**:
- 是否有更简单的实现方式？
- 是否有重复代码可以提取？
- 是否有不必要的复杂度？
- 是否遵循现有代码模式？

**违反示例**:
- 200 行的函数
- 复制粘贴的重复代码
- 晦涩难懂的变量名（如 `a`、`b`、`temp1`）

---

### X. 向后兼容 (Backward Compatibility)

**公共 API 和序列化格式必须保持向后兼容**

- 破坏性变更必须有版本迁移方案
- 旧场景文件必须能在新版本加载
- API 废弃必须有过渡期
- 版本号遵循 SemVer 规范

**版本策略**:
- MAJOR: 破坏性变更
- MINOR: 新功能（向后兼容）
- PATCH: Bug 修复（向后兼容）

**违反示例**:
- 修改序列化格式导致旧文件无法加载
- 删除公共 API 而无替代方案
- 不遵循 SemVer 的版本号

## 附加约束

### 安全要求

- 用户输入必须进行验证
- 文件操作必须限制在允许的目录内
- 外部资源加载必须有白名单机制
- 禁止 eval 和 Function 构造函数

### 可访问性

- 所有交互元素必须支持键盘操作
- 颜色对比度必须符合 WCAG 2.1 AA
- 重要操作必须有确认机制
- 错误提示必须清晰可读

### 国际化准备

- 用户可见文本不得硬编码
- 日期、数字格式必须使用库函数
- 为未来 i18n 预留结构

## 开发工作流

### 代码提交

- 提交信息格式：`<type>: <description>`
- Type 包括：feat、fix、docs、style、refactor、test、chore
- 示例：`feat: 添加材质缓存机制`

### 代码审查

- 所有代码必须经过审查
- 审查人必须检查宪法合规性
- 违反宪法的代码不得合并

### 测试要求

- 核心功能必须有手动测试清单
- 破坏性变更必须有回归测试
- 性能优化必须有前后对比数据

## 合规性检查

### 自查清单

开发过程中定期对照以下问题：

- [ ] 我的代码是否符合模块化原则？
- [ ] 是否使用了 Composition API？
- [ ] 模块间是否通过事件解耦？
- [ ] 序列化是否正确处理？
- [ ] 是否考虑了性能影响？
- [ ] 是否更新了相关文档？
- [ ] 代码是否清晰可维护？
- [ ] 是否保持了向后兼容？

### 审查清单

代码审查时必须检查：

- [ ] 宪法原则合规性
- [ ] 代码质量
- [ ] 文档完整性
- [ ] 测试覆盖
- [ ] 性能影响
